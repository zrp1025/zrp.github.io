JavaScript的代码规范和代码风格
发表于 2020-04-26| 更新于 2020-04-28|技术
阅读量:
程序是写给人读的，只是有时候让计算机执行以下。—— Donald Knuth

一、代码的风格约定

曾经在某综艺看到这样一个游戏，一群人排队轮流到同一块空白的黑板上画画，每人画一笔。最终出来的结果总是乱的一团糟。那如果在开始画之前，大家共同约定了画画的风格和规范，依照约定好的方式进行画画，最终的结果将会会有所改善。

在研发团队中，尽早的统一编程风格是非常重要的，有一下几个好处：

  • 节约时间：在打开有一个文件的时候，便可开始工作，而不再需要先去调整代码的缩进，因为所有的代码排版格式看起来非常一致，当项目非常庞大时，效果更加明显。
  • 容易识别代码错误：在统一了代码风格之后，当出现一段代码与众不同时，通常很大可能问题产生在这段代码中。

1.1 基本的格式化

1.1.1 缩进

每一行的层级由4个空格组成，避免使用制表符（Tab）进行缩进。（因为不同系统不同编译器的制表符占位约定存在差异。）

1.1.2 语句结尾

语句必须以分号结束。

1.1.3 行的长度

每行长度不应该超过80个字符。如果多余80个字符，应当在一个运算符（逗号，加号）后换行。下一行应当增加两级缩进（8个字符）。

1.1.4 空行

没有语义关联的代码之间用空行隔开。

  • 在方法之间
  • 在局部变量和第一条语句之间
  • 多行或单行注释之前
  • 在方法内逻辑片段之间插入空行，提高可读性

1.1.5 命名

  • 变量名和函数名采用小驼峰，变量名用前缀用名词，函数名前缀用动词
  • 变量名尽量语义体现数据类型，例如count、length、size表明用数字，name、title、message表明用字符串
  • 函数名动词前缀体现返回数据类型
  • 常量用大写和下划线组成
  • 构造函数用大驼峰，区分普通函数，当代码中使用大驼峰名词时，前面一定会有new运算符
动词 含义
can 函数返回一个布尔值
has 函数返回一个布尔值
is 函数返回一个布尔值
get 函数返回一个非布尔值
set 函数用来保存一个值

1.1.6 直接量

  1. 字符串: 用双引号将字符串括起来
  2. 数字
    1. 小数点前后部分不能省略
    2. 010不是10,是八进制中的8,为了避免误解,也因为很少使用八进制直接量,所以约定禁止使用
  3. null: 只在预设变量未来会被赋值为一个对象时使用
  4. undefined: 禁止使用undefined
  5. 对象: 创建对象使用对象直接量
  6. 数组: 创建数组使用数组直接量

1.2 如何写注释

注释也是为了节约时间成本,适度的注释用来解释代码的来龙去脉,在其他开发者就不需要从头看代码,只需要读取部分代码便可了解。

1.2.1 使用注释

  • 当出现难以理解的代码时，需要通过注释描述代码用途，帮助别人理解
  • 当出现可能被误认为错误的代码时，添加注释解释，避免被好心人错误修复
  • 文档注释，例如通过JSDoc Toolkit工具可以根据某些规则的注释来生成文档（后续另写文章介绍）

1.2.2 单行注释

  • 用于解释下一行代码，不能超过行字数约定，缩进与下一行代码一致
  • 用于注释大段代码

1.2.3 多行注释

  • 用于更详细的描述，注意格式

1.3 变量、函数、运算符

  • 用let取代var，禁止使用var。
  • 常量用const声明
  • 函数之间空行
  • 不使用全局的严格模式，在函数内部使用
  • 使用===和!==,禁止使用==和!=
  • 静止使用with,避免使用eval()
